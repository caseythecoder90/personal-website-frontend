1

Hero feels short next to other pages

Projects/Blog heroes have depth (tagline + meta + accent). Certifications has title + one subtitle and then drops straight into filters. It reads as "stub" compared to the rest of the site.

FIX · Add a small stat strip: "12 EARNED · 3 IN PROGRESS · 4 ORGS" directly below the subtitle. Keeps hero rhythm consistent with Projects.

2

Featured card is quiet

The star-shaped watermark is a generic "featured" signal and doesn't relate to the cert itself. On AWS, a viewer expects AWS iconography or the actual badge.

FIX · Use badgeUrl from the API as the watermark (or an organization mark). Falls back to the star if missing. Much richer for free — your backend already has the field.

3

Cards are dead-ends

Nothing is clickable. Viewers interested in a credential (hiring managers are!) can't verify it. You already have credentialUrl and slug in the response.

FIX · Two actions per card — View details (internal, if you build the detail route) and Verify ↗ (opens credentialUrl). Link is the biggest quality-of-life bump here.

4

No sense of time or progress

Issue dates are buried as inline text. A recruiter scanning wants "this is fresh" signal. Also, "In Progress" cards have nothing showing how in progress.

FIX · For earned: show year prominently (right-aligned, monospace). For in-progress: a small "Target Q2 '26" chip. Subtle, no bars.

5

Expired is hidden behind a filter that doesn't exist

Your backend supports EXPIRED status, the badge is styled for it, but the filter tabs only offer All / Earned / In Progress. Expired certs render in "All" but there's no way to see just the historical set. Minor but inconsistent.

FIX · Either add an "Expired" tab for completeness, or hide expired from "All" by default and remove EXPIRED from the filter array to tighten the promise. I'd vote for hide from All + separate "Archive" tab — it keeps the main grid feeling current.

YOUR QUESTION · BACKEND

Should you build a certification detail API?

Short answer: not for its own sake. You already return the full CertificationResponse from GET /certifications/slug/{slug} — everything needed for a detail page is already there. The question is whether a detail page adds value.

Build it if you will add…

  • A long-form "what I learned" write-up per cert
  • A gallery — study notes, lab screenshots, project links
  • Related blog posts (filtered by shared tech tags)
  • Embedded verification widget / downloadable PDF cert

Skip it if…

  • You'd just be expanding the description
  • No plan for supporting content beyond what the card shows
  • The credentialUrl ↗ is the real "more info" destination

MY TAKE · Skip a dedicated detail page for now. Add the Verify ↗ external link (which you already have data for) and a "Learnings" paragraph that expands inline on card click. Saves you a page build and reduces the depth of the nav tree.

Proposed refresh

All 5 upgrades applied. Same grid system, same tokens — just tighter information density and one new interaction pattern.

A

Missing: inline validation + field states

Backend rules: email is required + valid format, message min 20 / max 2000, name + subject max 255. The user needs to see these before hitting submit — a 400 from the server on submit feels broken.

FIX · Live validation on blur. Show min/max characters for the message. Error styling for error state (border-error, tiny helper text).

B

Missing: submission states

Design shows only the resting state. You need loading ("Sending…" + disable the button), success ("Got it — I'll reply within 48h"), and error ("Something broke, try again or email directly").

FIX · Replace the whole form card with a success panel on 200, with a reset CTA. Mock included below.

C

Missing: spam floor

Backend has IP rate-limiting which is a start, but a public POST endpoint without a honeypot or challenge will eat bot submissions. Cheap fix: hidden honeypot field that bots auto-fill.

FIX · <input name="website" class="sr-only" tabindex="-1"> + reject server-side if non-empty. Adds zero UX friction.

D

The stock office photo undercuts the brand

The rest of the site is editorial and considered. A desaturated generic workspace photo in the contact page feels like placeholder content that shipped.

FIX · Either a real photo of you, or kill the image and replace with a timezone/availability panel: "Based in [City] · UTC-5 · Currently accepting work". The second option is stronger.

E

Minor: social links should show the actual handle

"Email / GitHub / LinkedIn" with nothing after is a vague promise. Preview the handle or address underneath the label — [email protected], github.com/caseyquinn. Reduces a click to copy or confirm and signals that you actually own the accounts (not just decorative icons).

Proposed: three states side-by-side

Same layout, same tokens — proper feedback loops, an availability panel replacing the stock photo, and a visible success state.

03 · RESUME

One button vs. a whole page.

Recommendation

Build the page. Make nav work both ways.

YOUR QUESTION

"In the design there is just a resume button which should download my resume. Should I also have a resume page where they can view a resume in my design style and then also download a PDF?"

Yes — and here's why the page is actually the higher-leverage choice.

ARGUMENT

The PDF is the "I already decided" artifact

A recruiter who wants a PDF wants it because their ATS demands it. They're not comparing you to other candidates at this moment — they're filing you. Good, serve it fast.

ARGUMENT

The page is where discovery happens

A hiring manager browsing your site is in evaluation mode. A beautifully typeset resume page that matches your visual language signals "this person cares about craft" in a way a PDF literally cannot — PDFs always look like a PDF.

ARGUMENT

Resumes go stale; pages don't

Your site is already data-driven. A resume page reuses project + certification + technology data you already publish. Update once, it reflects everywhere. PDF becomes the snapshot export, not the source of truth.

The navbar Resume button: split it.

Right now you have one button doing one thing. Replace it with a split-action control:

Resume download

PATTERN · "Resume" → /resume page · ↓ icon → GET /resume/download directly · tooltip on download icon: "Download PDF".

Both audiences served in the same footprint. The default (big label) is the more interesting click for browsing users; the small icon is the power-user shortcut for anyone who knows what they want.

Proposed: /resume page

Editorial layout. Two columns on desktop: narrow-left identity rail, wide-right content. Same type + color system as the rest of the site. Sticky download button. Prints clean if anyone actually hits Cmd-P.

BACKEND · WHAT YOU HAVE

// Already exposed
GET /api/v1/resume → metadata
GET /api/v1/resume/download → 302 to PDF

ResumeResponse already returns filename, size, uploadedAt. Use uploadedAt as the "Last updated" stamp in the UI — it's a trust signal.

BACKEND · WHAT TO ADD

The /resume page is just composition of existing endpoints. You don't need a new API for it:

  • /resume for metadata
  • /certifications/published
  • /projects/featured
  • /technologies/featured

The "experience" / "about" / "education" blocks are the only net-new content. Store as JSON in a config file, or add a small /profile endpoint later if you want admin editing.

  • Inline-expand "Learnings" on cert cards. Avoids building a detail page.
  • Add count pills to filter tabs. Works on /blog too. Small pattern investment.

🟣 THE BIG ONE

Build /resume as a first-class page.

It's the single highest-leverage add. It composes data you already have, reinforces your brand precisely where it matters most (someone evaluating you as a candidate), gives power-users the PDF shortcut they want, and gives browsers a rich artifact they can actually read on-screen. A PDF alone is half the story.
